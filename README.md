# Syntax: A Comprehensive Android Event Management Application

Syntax is a fully featured Android application developed as a six person group project for a software engineering course. The application enables users to discover, join, and manage events through a sophisticated multi role system that supports entrants, organizers, co organizers, and administrators. Built with Android's modern development stack including fragments, navigation components, and Material Design principles, SyntaxEvents demonstrates a complete event lifecycle from creation through attendee management and post event analysis.

The application implements a clean architecture using the repository pattern and MVVM design principles. Firebase serves as the complete backend solution with Firestore handling structured data like user profiles, events, comments, and notifications, while Realtime Database manages binary content such as event posters and QR codes. Anonymous authentication allows users to immediately access the app without barriers, with profile creation handled through an onboarding flow that captures names, emails, phone numbers, and role preferences.

Entrants experience a rich feature set starting with a home screen that displays all open events they have not yet joined. Events are filterable by date range, capacity, and keyword search. When an entrant finds an event of interest, they can view detailed information including the event poster, full description, date range, registration period, waitlist capacity, and lottery criteria formatted as bullet points. The event detail screen includes a complete commenting system where users can post comments, delete their own comments, and report inappropriate content from others. The reporting system tracks which users have reported each comment to prevent duplicate reports and provides administrators with moderation tools.

The join mechanism incorporates geolocation validation for events that require it. When an entrant attempts to join an event that has geolocation enabled, the application checks the user's current location against the event's city using reverse geocoding. If the user is not within the required geographic area, the join operation is rejected with a clear explanation. This feature ensures that events requiring physical attendance only accept participants who are genuinely able to attend. For events that do not require geolocation, the join process simply adds the user to the waitlist and increments the event's counter using Firestore transactions to ensure data consistency.

The lottery system represents one of the most technically complex features. Organizers can run a lottery at any time by specifying a sample size. The system randomly selects winners from the waitlist, excluding users who have already received invitations. For each selected winner, the system creates an invitation document in Firestore and adds the user to the event's invitedUserIds list. For users not selected, a separate notification is created to inform them they were not chosen. When a winner accepts the invitation, they are moved to the selected entrants subcollection, while declines trigger a replacement drawing using the same random selection logic. This entire workflow is managed through asynchronous callbacks with proper error handling and atomic operations.

Organizers have access to a comprehensive event management dashboard. They can create events through a multi step form that captures name, description, location, capacity, event dates, registration period dates, lottery criteria, and geolocation requirements. The location field includes an autocomplete feature powered by a CSV file containing world cities with over forty thousand entries, providing real time suggestions as the user types. The lottery criteria field uses a custom bullet point helper that automatically formats new lines with bullet symbols and prevents deletion of the initial bullet point, making it easy for organizers to create readable lists of requirements.

After creating the event, organizers proceed to the poster upload screen where they can select an image from their device gallery. The selected image is resized to a maximum dimension of five hundred twelve pixels and compressed at sixty percent quality before being base64 encoded and stored in Firebase Realtime Database. The same image is cached in memory using ImageCacheManager, a singleton that maintains both bitmap and base64 caches keyed by event ID, enabling instant loading when users revisit the event. The final step generates a QR code containing the event ID using the ZXing library, displays it to the organizer for download, and stores the base64 encoded version in Realtime Database for later retrieval by entrants who scan the code.

The notification system implements a fan out architecture for bulk messaging. When an organizer sends a notification to a group of entrants, the system first creates a canonical record in the notifications collection for audit purposes. It then queries the appropriate subcollection for the event, which could be waitlist entrants, selected entrants, or cancelled entrants. For each unique user in the target groups, the system checks their notification preferences stored in their profile. Users who have opted out of organizer notifications are skipped. A personal copy of the notification is then written to that user's notifications subcollection under users slash userId slash notifications. This approach ensures that each user's notification inbox is independent and that deleted users do not affect historical data. The same system supports administrator broadcasts to all users, with the special event identifier set to ADMINISTRATION to indicate platform level messages.

The comment system includes pagination to improve performance when events accumulate many comments. Comments are loaded ten at a time with next and previous page buttons, and the adapter dynamically updates role based action buttons. Regular users see a report button on comments from other users, which becomes disabled after reporting and changes to a reported label. The comment author sees a delete button instead of report. Organizers and administrators see delete buttons on all comments regardless of authorship. The report count is displayed as a badge on each comment in the admin browse screen, and administrators can filter to view only reported comments.

The admin dashboard provides complete oversight of the entire system. Administrators can browse all events with real time search by name, description, or location, and click any event to view its details with options to delete it. The event deletion process removes the Firestore document and also cleans up associated media from Realtime Database. For user management, administrators see a combined list of active profiles from the profiles collection and deleted profiles from the deleted profiles collection, each marked with appropriate status indicators. Clicking a profile reveals detailed information including name, email, device ID, phone number, and role toggles. Administrators can add or remove roles from any user with safety checks preventing removal of the last remaining role. When a user is deleted, their profile is first archived to the deleted profiles collection with a timestamp before being removed from active profiles, preserving audit history.

The co organizer feature extends event management capabilities by allowing organizers to delegate responsibilities. Through the notification screen, organizers can search for entrants by name, email, or phone number and invite them to become co organizers. The invitation system creates a notification for the recipient, and upon acceptance, the user is added to the event's co organizers subcollection. Co organizers have the same management permissions as the original organizer, enabling collaborative event administration without granting full organizer status across all events.

The map integration uses OSMDroid to display attendee locations for events that have geolocation enabled. When entrants join a geolocation event, their latitude and longitude are stored in the waitlist entrants document. The organizer can view a map showing pins for each attendee at their approximate location, with the pin labels showing the attendee's name. This feature provides valuable insights for organizers about the geographic distribution of their participants.

The profile system allows users to edit their personal information and manage notification preferences. Users can choose to receive or block notifications from organizers and administrators independently, giving them granular control over their notification experience. Account deletion is supported with confirmation dialogs and archive before delete semantics to prevent accidental data loss.

Throughout the application, careful attention has been paid to user experience with entrance animations, loading states, empty states, and error handling. Card views animate in with staggered delays, creating a polished feel. Loading spinners indicate when data is being fetched, and empty states provide helpful messages when no data is available. Form validation includes checks for required fields, email format validation, and date range validation. All network operations include error handling with user friendly messages.

From a technical perspective, SyntaxEvents demonstrates proficiency with Android development best practices including fragment lifecycle management, navigation component with safe args, ViewModel for configuration change resilience, LiveData for reactive UI updates, RecyclerView with multiple view types, custom adapters, background threading for image processing, and Firebase integration at an advanced level. The codebase is well organized with clear separation of concerns between fragments, repositories, models, and adapters. Each class includes comprehensive JavaDoc documentation describing its purpose and usage.

This project represents a complete full stack Android application that could realistically be deployed as a production event management platform. It showcases the ability to design and implement complex multi user systems with role based access control, real time data synchronization, media handling, and sophisticated business logic like lottery systems and geolocation validation. For employers reviewing this project, it demonstrates practical experience with Firebase, Android architecture components, Material Design, and the ability to work within a team to deliver a working product with substantial feature scope.
